When and why to use AmpersandJS

Posted on 2015-07-12   |  

To build the project I’ve been working on for quit some time, we’ve crossed a lot of deep seas in search of the ideal framework. The more frameworks we tried, the more often we started over again.

At the start, we were convinced that a full-featured CMS-framework could serve all our needs. Several months later, we concluded that the only thing we were actually using, was the user-management part of the open-sourced bulky system. Everything else was custom build on top of this framework and … unfortunately … we had ourselved “vendor-locked” whith this open-source system.

Our needs…

We are working on a Web App which feels like a native app. Our app has a need for multiple pages. The content on some pages needs to react on user input or is dynamically generated based on some models. Other pages are lists of multiple models of certain types… And on top of all of this, everything needs to run offline.

1. Multiple pages (a Router)

The absolute core-business our app needs to do, is to open up different pages, or as it’s called in the web-app-world: views.

2. Models and Collections

Our app needs to show some stuff to user. In the case of neoScores, we need to show a piece of Sheet Music. What is a score, a piece of Sheet Music? It’s a combination of a title, composer, copyrights, (…), and of course a bunch of staffs, notes, bars, accidentals and a lot, lot more. So we need to be able to bundle all these properties into something we can manage easily while being flexible to change some of the properties via some user input tools.
All of these properties together form our “Model”.

On the other hand, we need to be able to list up all the scores for one user into the user’s library. The user needs to be able to make subsections of his library (“Playlist” or “Setlists”) or to change some properties of all our models together at once via some user input tools. A “Collection” is exactly what can do all of this.

3. Offline Use

Our app is build on top of the concept off “Offline First” which means that LOCAL (in-browser) databases (“indexDB”,“WebSQL”,“LocalStorage”) must be addressed to look for the requested data, in combination with our ONLINE databases, to synchronise everything with the cloud, in real time, over multiple accounts, devices and platforms at once.

What about Object.observe?

In future the javascript native feature of “Object.observe” is exactly what does all of this (and more), and it will be supported natively in all major browsers. But… it’s syntax is more complex and used in a big project, some sort of framework needs to be written arround it. AmpersandJS does it in an easy way, and probably will serve as a polyfill soon whenever your browser supports it.

The good part of AmpersandJS

All the little pieces of code of AmpersandJS are build to do just one thing.

  • creation of models (ampersand-state)
  • creation of collections (of these models) (ampersand-collection)
  • showing the models (or collections) via the DOM, and keep all values of all properties up to date in real time (ampersand-view)
  • managing url’s to execute different code (ampersand-router)
  • … code which visualises different views (ampersand-view-switcher)

And you can require all of these little pieces of code:

  • individually,
  • whenever you need them,
  • independantly from each other,
  • independantly from other frameworks,
  • mixed together with other frameworks,
  • …

… so you end up with a build-file which includes ONLY the code you actually use!!!

TL;DR

When you start building a new project, lot’s of the time you start scafolding multiple (micro-) frameworks together and end up with a lot of custom scripts to bind one and another together. This often leads to your own bootstrapping-code which more and more becomes to grow and start looking as your own framework.

If this sounds familiar, have a look at AmpersandJS.
It’s:

  • Lightweight
  • Performant
  • Easy to learn
  • Selfexplenatory
  • Independant but compatible with any other frameworks
  • Opensource and easily accessible to start your own contribution

AmpersandJS gives you the standard concepts of the other big frameworks like Angular or Backbone (Models, Collections and Views), but it’s all conceived as a combination of micro-frameworks. In other words, you only need to require just that little bit of code you actually use, without having to include a bulky, fully fledged, big ass framework, filled with features you’ll never use.
